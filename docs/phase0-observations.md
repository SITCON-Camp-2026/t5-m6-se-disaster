# Phase 0 Observations

## 我們做出了什麼？

- 補上了第一階段的整理工作台，從 `App.tsx` 首頁可以直接進入。
- 先顯示原始資訊清單，並提供每筆資料的來源、查核狀態與更新時間。
- 實作可編輯草稿範例，使用者可以儲存、重設、刪除草稿。

## 哪些資料讓 agent 很容易整理？

- `M-001`、`M-003`、`M-004` 這類較明確描述現場現況的回報，比較容易判斷是否為「物資需補充」或「現場狀態更新」。
- 如果原文明確說明仍缺雨鞋、缺水或現場集合方式，就可以先用作候選判斷。

## 哪些資料讓 agent 亂補或誤判？

- `M-005` 只有截圖說「中午前道路封閉」，但沒有確認時間和來源，不能直接當成官方路況公告。
- `M-012` 由外地家屬轉述，位置與需求都不在現場，這種資訊容易被誤判為「直接派人處理」。

## 哪些資訊不能直接變成任務？

- `M-001` 的「老雜貨店後面」不是精確地址，還需要人工確認位置是否安全。
- `M-006` 有人說學校側門適合作為集合點，但也有志工說剛剛淹水，不能直接派人前往。
- `M-009` 只說臨時集合點仍開放，但限制只有已報到的清淤志工，這不能直接轉成一般物資運送任務。

## 哪些資訊的操作者不是當事人？

- `M-012` 由外地家屬來電轉述，資訊來源不是現場本人。
- `M-011` 由志工代長者轉述，長者本人是否同意公開位置還沒確認。
- `M-004` 和 `M-007` 來自社群貼文，貼文者與現場實際狀況可能有落差。

## 我們剛剛偷偷假設了什麼？

- 我們沒有假設未確認資訊是正確的；這個工作台只展示安全預設，並提醒「請再確認」。
- 我們假設 `sourceType` 只代表來源類別，不等於已查核。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 確認每筆資訊是否有原始來源連結或拍照證據。
- 讓後續協作者補上「哪些判斷是 agent 建議、哪些是人類已確認」的欄位。
- 釐清「現場回報」「社群貼文」「官方公告」之間的可信等級。

## 我們需要更清楚資料格式的地方

- 需要明確欄位區分「任務需求」「物資狀態」「集合點資訊」「危險狀況」。
- 需要更清楚 `verificationStatus` 的定義和可能值。

## 我們需要人工確認的地方

- 所有 `needs_review` 和 `unverified` 的資料都需要人類確認，特別是 `M-005`、`M-011`、`M-012`。
- 需要確認 `M-006` 的集合點是否安全，以及 `M-009` 的限制是否仍然有效。
